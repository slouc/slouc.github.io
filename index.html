We can implement both apply() and product() with monad’s operations unit(), map() and flatten().

Given:

trait Monad[F[_]] {
  def map[A, B](fa: F[A], f: A => B): F[B] = ???
  def flatten[A](fa: F[F[A]]): F[A] = ???
}

they can be implemented as:

trait Monad2[F[_]] extends Monad[F] {
  def apply[A, B](fa: F[A], fab: F[A => B]): F[B] = flatten(map(fab, (f: A => B) => map(fa, f)))
  def product[A, B](fa: F[A], fb: F[B]): F[(A, B)] = flatten(map(fa, (a: A) => map(fb, (b: B) => (a,b))))
}

Here’s an example of using apply() with a curried function. I will provide a concrete implementation of apply() for List just for simplicity.

def apply[A, B](list: List[A], f: List[A => B]): List[B] = f.flatMap(f => list.map(f))

val curriedF = (a: Int) => (b: Int) => (c: Int) => a + b + c

val list1 = List(1)
val list2 = List(2)
val list3 = List(3)
val result1 = apply(list1, List(curriedF))
val result2 = apply(list2, result1)
val result3 = apply(list3, result2)

println(result3) // List(6)

The usage is the same as in applicative case. However, there’s a significant difference - if apply (or product, for that matter) is implemented using a flatMap, we are taking advantage of monad’s power to chain things into series. Standard apply/product from applicative lack this ability.

Since we worked with apply() in the previous example, let’s now take product(). We can implement it once with monad’s functions, and once in a typical applicative fashion (by using zip):

// we'll need these throughout the example
import scala.concurrent.{Await, Future}
import scala.concurrent.duration._
import scala.concurrent.ExecutionContext.Implicits.global

def product[A, B](f1: => Future[A], f2: => Future[B]): Future[(A, B)] = f1.flatMap(a => f2.map(b => (a,b)))
def product2[A, B](f1: => Future[A], f2: => Future[B]): Future[(A, B)] = f1 zip f2

So, second one is what you get when you’re using applicative, while first is the advanced, monad version. Note that both are taking their parameters by-name instead of by-value in order to prevent futures from executing as soon as they are passed.

Let’s now create the futures:

def first = Future { println("first"); Thread.sleep(2000); 1 }
def second = Future { println("second"); Thread.sleep(2000); 2 }

Using serial product:

// prints "first", then two seconds later "second"
val result1 = product(first, second).map { case (a, b) => a + b }
Await.result(result1, 10 seconds)

Using parallel product:

// performs in parallel; order of printing "first" and "second" is indeterministic
val result2 = product2(first, second).map { case (a, b) => a + b }
Await.result(result2, 10 seconds)

Finally, the scalaz version. I never managed to get “second” printed before “first”, but they are still almost simultaneous (unlike the first product() where we can clearly see the two second difference between each printout).

import scalaz._, Scalaz._

// "first" and "second" are printed simultaneously
val result3 = (first |@| second) { case (a, b) => a + b }
Await.result(result3, 10 seconds)

While monad could have methods such as apply() or product() exposed in its API, it probably doesn’t make a lot of sense. Still, if one needs them, we have seen here that they can be easily implemented in terms of existing monad functions. Note that even if their signatures are the same as applicative’s, we have also seen that monad’s power - ability to chain computations in series - is still present. We could also make them work in parallel; in this particular example we simply need to go from by-name to by-value invocation by removing the => from each parameter type. But if one needs a parallel computation, best practice is to use the least powerful abstraction that does the job, instead of “dumbing down” the monad.
